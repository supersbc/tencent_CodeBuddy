## 📊 第一部分：业务基础信息

### 1.1 行业与场景

#### `industry` - 行业类型 ⭐⭐⭐⭐⭐
- **类型**: 选择框
- **必填**: 是
- **说明**: 选择所属行业，影响架构模式和性能优化策略
- **选项**:
  - **金融行业**: 银行、证券、保险、支付
  - **电商行业**: 电商平台、O2O、社交电商、跨境电商
  - **游戏行业**: 手游、端游、页游、H5游戏
  - **互联网**: 社交网络、视频平台、新闻资讯
  - **其他**: 政务、医疗、教育、物流、新零售、物联网、通用
- **影响**: 
  - 金融行业：强调高可用(99.99%+)、强一致性、合规要求
  - 电商行业：高并发(10万+QPS)、促销支持、读写分离
  - 游戏行业：低延迟(<10ms)、分区分服、实时同步

#### `business_type` - 业务类型 ⭐⭐⭐⭐⭐
- **类型**: 选择框
- **必填**: 是
- **说明**: 业务类型决定数据库优化方向
- **选项**:
  - `OLTP` - 在线事务处理（高并发、低延迟）
  - `OLAP` - 在线分析处理（复杂查询、大数据量）
  - `HTAP` - 混合事务分析处理（兼顾OLTP和OLAP）
  - `实时流处理` - 实时数据处理
  - `批处理` - 批量数据处理

#### `scenario_name` - 场景名称 ⭐⭐⭐⭐⭐
- **类型**: 文本框
- **必填**: 是
- **说明**: 具体业务场景，如：核心交易系统、订单系统、用户中心
- **示例**:
  - 金融：核心交易系统、支付系统、风控系统
  - 电商：订单系统、商品系统、库存系统
  - 游戏：玩家数据、排行榜、背包系统

### 1.2 业务规模

#### `user_count` - 用户数量 ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: 个
- **说明**: 总用户数，用于容量规划
- **参考值**:
  - 小型: < 10万
  - 中型: 10万 - 100万
  - 大型: 100万 - 1000万
  - 超大型: > 1000万

#### `daily_active_users` - 日活用户 ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: 个
- **说明**: 每日活跃用户数，影响并发设计
- **计算**: 通常为总用户数的 10%-30%

#### `peak_user_count` - 峰值用户数 ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: 个
- **说明**: 业务高峰期的同时在线用户数
- **计算**: 通常为日活的 20%-50%

### 1.3 业务特征

#### `has_seasonal_traffic` - 季节性流量 ⭐⭐⭐
- **类型**: 复选框
- **说明**: 是否有明显的季节性流量波动
- **示例**: 电商双11、春节红包、游戏节假日

#### `seasonal_peak_multiplier` - 季节性峰值倍数 ⭐⭐⭐
- **类型**: 数字
- **默认**: 1.0
- **说明**: 季节性高峰相对平时的流量倍数
- **参考值**:
  - 电商双11: 5-10倍
  - 春节红包: 3-5倍
  - 游戏节假日: 2-3倍

#### `has_promotion_scenario` - 促销场景 ⭐⭐⭐
- **类型**: 复选框
- **说明**: 是否有促销、秒杀等高并发场景
- **影响**: 需要特殊的架构设计和缓存策略

## 📊 第二部分：数据规模与特征

### 2.1 数据总量

#### `total_data_size_gb` - 当前数据总量(GB) ⭐⭐⭐⭐⭐
- **类型**: 数字
- **必填**: 是
- **单位**: GB
- **说明**: 当前数据库总大小，影响存储和性能配置
- **参考值**:
  - 小型: < 100GB
  - 中型: 100GB - 1TB
  - 大型: 1TB - 10TB
  - 超大型: > 10TB

#### `data_growth_rate_yearly` - 年度数据增长率(%) ⭐⭐⭐⭐⭐
- **类型**: 数字
- **必填**: 是
- **单位**: %
- **说明**: 年度数据增长百分比，用于容量规划
- **参考值**:
  - 稳定业务: 10%-20%
  - 成长业务: 30%-50%
  - 快速增长: 100%+

#### `expected_data_size_3year_tb` - 3年后预期数据量(TB) ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: TB
- **说明**: 3年后的预期数据量，用于长期规划
- **计算**: 当前数据量 × (1 + 增长率)³

### 2.2 数据库与表结构

#### `database_count` - 数据库数量 ⭐⭐⭐
- **类型**: 数字
- **默认**: 1
- **说明**: 实例中的数据库个数

#### `table_count` - 表总数 ⭐⭐⭐⭐⭐
- **类型**: 数字
- **必填**: 是
- **说明**: 数据库中的表总数
- **参考值**:
  - 小型: < 100张
  - 中型: 100-500张
  - 大型: 500-2000张
  - 超大型: > 2000张

#### `max_table_size_gb` - 最大单表大小(GB) ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: GB
- **说明**: 单表最大数据量，判断是否需要分表
- **建议**: 
  - > 100GB: 建议分表
  - > 500GB: 强烈建议分表

#### `max_table_rows` - 最大单表行数 ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: 行
- **说明**: 单表最大记录数
- **建议**:
  - > 1000万行: 考虑分表
  - > 5000万行: 建议分表
  - > 1亿行: 强烈建议分表

### 2.3 数据特征

#### `has_large_table` - 是否有大表 ⭐⭐⭐
- **类型**: 复选框
- **说明**: 是否有超过100GB的大表
- **影响**: 需要特殊的索引和查询优化

#### `has_blob_data` - 是否有BLOB数据 ⭐⭐⭐
- **类型**: 复选框
- **说明**: 是否存储图片、文件等二进制数据
- **建议**: 大量BLOB数据建议使用对象存储

#### `has_time_series_data` - 是否有时序数据 ⭐⭐⭐
- **类型**: 复选框
- **说明**: 是否有时间序列数据（如日志、监控数据）
- **影响**: 可能需要时序数据库或分区表

## 📊 第三部分：性能指标（核心）

### 3.1 QPS指标

#### `avg_qps` - 平均QPS ⭐⭐⭐⭐⭐
- **类型**: 数字
- **必填**: 是
- **单位**: 次/秒
- **说明**: 每秒查询数，核心性能指标
- **参考值**:
  - 低性能: < 1,000
  - 中性能: 1,000 - 10,000
  - 高性能: 10,000 - 50,000
  - 超高性能: 50,000 - 100,000
  - 极致性能: > 100,000

#### `peak_qps` - 峰值QPS ⭐⭐⭐⭐⭐
- **类型**: 数字
- **必填**: 是
- **单位**: 次/秒
- **说明**: 业务高峰期QPS，用于容量规划
- **计算**: 通常为平均QPS的 2-5倍

#### `read_write_ratio` - 读写比例 ⭐⭐⭐⭐⭐
- **类型**: 文本框
- **必填**: 是
- **格式**: "读:写"，如 "7:3"
- **说明**: 读操作和写操作的比例
- **参考值**:
  - 读多写少: 9:1, 8:2, 7:3（适合读写分离）
  - 读写均衡: 6:4, 5:5
  - 写多读少: 3:7, 2:8（需要优化写性能）

### 3.2 TPS指标

#### `avg_tps` - 平均TPS ⭐⭐⭐⭐⭐
- **类型**: 数字
- **必填**: 是
- **单位**: 次/秒
- **说明**: 每秒事务数，事务处理能力指标
- **参考值**:
  - 低性能: < 500
  - 中性能: 500 - 5,000
  - 高性能: 5,000 - 20,000
  - 超高性能: > 20,000

#### `peak_tps` - 峰值TPS ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: 次/秒
- **说明**: 业务高峰期TPS
- **计算**: 通常为平均TPS的 2-5倍

### 3.3 并发指标

#### `concurrent_connections` - 并发连接数 ⭐⭐⭐⭐⭐
- **类型**: 数字
- **必填**: 是
- **单位**: 个
- **说明**: 平均同时连接数，影响连接池配置
- **参考值**:
  - 小型: < 100
  - 中型: 100 - 1,000
  - 大型: 1,000 - 5,000
  - 超大型: > 5,000

#### `max_connections` - 最大连接数 ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: 个
- **说明**: 峰值时的最大连接数
- **计算**: 通常为并发连接数的 1.5-2倍

### 3.4 响应时间要求

#### `avg_response_time_ms` - 平均响应时间(ms) ⭐⭐⭐⭐⭐
- **类型**: 数字
- **必填**: 是
- **单位**: 毫秒
- **说明**: SQL平均执行时间，用户体验关键指标
- **参考值**:
  - 优秀: < 10ms
  - 良好: 10-50ms
  - 一般: 50-100ms
  - 较差: > 100ms

#### `p99_response_time_ms` - P99响应时间(ms) ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: 毫秒
- **说明**: 99%请求的响应时间
- **参考值**:
  - 优秀: < 50ms
  - 良好: 50-100ms
  - 一般: 100-200ms
  - 较差: > 200ms

#### `max_acceptable_latency_ms` - 最大可接受延迟(ms) ⭐⭐⭐⭐
- **类型**: 数字
- **默认**: 100
- **单位**: 毫秒
- **说明**: 业务可接受的最大延迟
- **行业参考**:
  - 金融交易: < 50ms
  - 电商下单: < 100ms
  - 游戏操作: < 10ms
  - 内容浏览: < 200ms

## 📊 第四部分：高可用与容灾

### 4.1 可用性要求

#### `availability_level` - 可用性级别 ⭐⭐⭐⭐⭐
- **类型**: 选择框
- **必填**: 是
- **选项**:
  - `基础(99%)` - 年停机时间 3.65天
  - `标准(99.9%)` - 年停机时间 8.76小时
  - `高可用(99.99%)` - 年停机时间 52.56分钟
  - `关键(99.999%)` - 年停机时间 5.26分钟
  - `极致(99.9999%)` - 年停机时间 31.5秒
- **行业参考**:
  - 金融核心: 99.99% - 99.999%
  - 电商平台: 99.9% - 99.99%
  - 游戏服务: 99.9%
  - 企业应用: 99% - 99.9%

#### `need_high_availability` - 需要高可用 ⭐⭐⭐⭐⭐
- **类型**: 复选框
- **必填**: 是
- **说明**: 是否需要主从/多主架构
- **影响**: 
  - 是：配置主从架构，自动故障切换
  - 否：单机部署，成本较低

### 4.2 高可用方案

#### `failover_time_seconds` - 故障切换时间(秒) ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: 秒
- **说明**: 自动故障切换耗时
- **参考值**:
  - 快速: < 10秒
  - 标准: 10-30秒
  - 较慢: > 30秒

#### `auto_failover` - 自动故障切换 ⭐⭐⭐⭐
- **类型**: 复选框
- **说明**: 是否支持自动故障切换
- **建议**: 高可用场景强烈建议开启

### 4.3 灾备要求

#### `need_disaster_recovery` - 需要灾备 ⭐⭐⭐⭐
- **类型**: 复选框
- **说明**: 是否需要异地灾备
- **行业要求**:
  - 金融：强制要求
  - 电商：建议配置
  - 游戏：可选

#### `dr_type` - 灾备类型 ⭐⭐⭐⭐
- **类型**: 选择框
- **选项**:
  - `无` - 不配置灾备
  - `同城双活` - 同城两个数据中心
  - `异地灾备` - 异地备份中心
  - `两地三中心` - 本地+同城+异地
  - `多地多活` - 多个地域同时提供服务

#### `rto_minutes` - RTO恢复时间目标(分钟) ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: 分钟
- **说明**: 故障后多久恢复服务
- **参考值**:
  - 关键业务: < 5分钟
  - 重要业务: 5-30分钟
  - 一般业务: 30-60分钟

#### `rpo_minutes` - RPO恢复点目标(分钟) ⭐⭐⭐⭐
- **类型**: 数字
- **单位**: 分钟
- **说明**: 可接受的数据丢失时间
- **参考值**:
  - 零丢失: 0分钟（同步复制）
  - 极少丢失: < 5分钟
  - 可接受: 5-60分钟

- **单位**: 个
- **说明**: 读从库的数量

#### `expected_shard_count` - 预期分片数 ⭐⭐⭐⭐

- **单位**: 个
- **说明**: 建议的分片数量
- **参考值**:
  - 小规模: 2-4片
  - 中规模: 4-8片
  - 大规模: 8-16片
  - 超大规模: 16-32片

## 📈 参数对预测的影响

### 影响架构类型的关键参数

| 参数 | 影响 |
|------|------|
| `total_data_size_gb` | 数据量大 → 分布式架构 |
| `avg_qps` / `peak_qps` | QPS高 → 分布式/读写分离 |
| `need_high_availability` | 是 → 主从/多主架构 |
| `need_sharding` | 是 → 分库分表架构 |
| `industry` | 金融 → 高可用架构 |

### 影响节点数量的关键参数

| 参数 | 影响 |
|------|------|
| `peak_qps` | QPS越高 → 节点越多 |
| `total_data_size_gb` | 数据量越大 → 节点越多 |
| `need_disaster_recovery` | 是 → 增加灾备节点 |
| `read_write_ratio` | 读多 → 增加从库节点 |
| `expected_shard_count` | 分片数 → 节点数 |

### 影响成本的关键参数

| 参数 | 影响 |
|------|------|
| `total_data_size_gb` | 数据量 → 存储成本 |
| `peak_qps` | 性能要求 → 硬件成本 |
| `availability_level` | 可用性 → 冗余成本 |
| `need_disaster_recovery` | 灾备 → 额外成本 |
| `need_encryption` | 加密 → 性能和成本 |
